# AI会议助手代码规范和开发流程

## 1. 文档概述

### 1.1 文档目的
本文档定义了AI会议助手项目的代码规范和开发流程，旨在确保代码质量，提高代码可维护性和可读性，促进团队协作，以及确保开发流程的高效和规范。本文档为开发团队提供了明确的编码指导和开发流程规范，确保开发工作的一致性和有效性。

### 1.2 适用范围
本文档适用于AI会议助手项目的所有开发活动，包括前端开发、后端开发、数据库开发和DevOps工作。

### 1.3 术语定义
| 术语 | 解释 |
|------|------|
| SOLID | 面向对象设计的五个原则：单一职责、开放封闭、里氏替换、接口隔离、依赖倒置 |
| KISS | 保持简单原则，避免不必要的复杂性 |
| DRY | 不要重复自己，避免代码冗余 |
| YAGNI | 你不会需要它，只实现当前所需的功能 |
| TypeScript | JavaScript的超集，添加了类型系统 |
| ESLint | 代码质量检查工具 |
| Prettier | 代码格式化工具 |
| Conventional Commits | 规范的提交信息格式 |
| CI/CD | 持续集成/持续部署 |
| PR | Pull Request，代码合并请求 |
| Code Review | 代码评审 |

## 2. 代码规范

### 2.1 通用规范

#### 2.1.1 命名规范
- **变量命名**：使用小驼峰命名法，变量名应清晰表达其用途
  - 示例：`userName`, `meetingTime`, `isActive`
- **函数命名**：使用小驼峰命名法，函数名应清晰表达其功能，动词开头
  - 示例：`getUserInfo()`, `createMeeting()`, `validateInput()`
- **类命名**：使用大驼峰命名法，类名应清晰表达其职责
  - 示例：`UserService`, `MeetingController`, `SpeechRecognizer`
- **常量命名**：使用全大写，单词间用下划线分隔
  - 示例：`MAX_RETRY_COUNT`, `API_BASE_URL`, `DATABASE_NAME`
- **文件命名**：使用小驼峰命名法，文件名应清晰表达其内容
  - 示例：`userService.ts`, `meetingList.vue`, `config.ts`
- **目录命名**：使用小驼峰命名法，目录名应清晰表达其用途
  - 示例：`utils`, `components`, `services`

#### 2.1.2 代码格式
- **缩进**：使用2个空格进行缩进
- **换行**：每行最多120个字符，超过则换行
- **空格**：
  - 操作符前后各加一个空格，如 `a + b`
  - 逗号后加一个空格，如 `[1, 2, 3]`
  - 分号后加一个空格（如果使用分号）
  - 括号内侧不加空格，如 `function() {`
- **空行**：
  - 函数、类、接口之间空两行
  - 代码块之间空一行
  - 同一函数内的不同逻辑块之间空一行
- **引号**：统一使用单引号 `'`，除非字符串中包含单引号

#### 2.1.3 注释规范
- **单行注释**：使用 `//`，注释内容与 `//` 之间加一个空格
  - 示例：`// 计算用户总数`
- **多行注释**：使用 `/* */`，适用于较长的注释
  - 示例：`/* 这是一个多行注释，用于描述复杂的逻辑 */`
- **JSDoc注释**：用于函数、类、接口等，描述其功能、参数、返回值等
  - 示例：
    ```typescript
    /**
     * 获取用户信息
     * @param userId 用户ID
     * @returns 用户信息
     */
    function getUserInfo(userId: string): User {
      // 实现逻辑
    }
    ```
- **注释原则**：
  - 注释应清晰、简洁，避免冗余
  - 注释应解释"为什么"，而不是"是什么"
  - 复杂逻辑必须添加注释
  - 公共API必须添加JSDoc注释
  - 定期清理过时的注释

#### 2.1.4 类型规范
- 尽可能使用具体的类型，避免使用 `any` 类型
- 为函数参数和返回值添加类型注解
- 为对象和数组添加类型定义
- 使用接口或类型别名定义复杂类型
- 优先使用 `interface` 定义对象类型，使用 `type` 定义联合类型、交叉类型等

#### 2.1.5 错误处理规范
- 使用 `try-catch` 捕获异常
- 抛出具体的错误类型，而不是简单的字符串
- 错误信息应清晰、具体，便于调试
- 记录错误日志，包括错误类型、错误信息、堆栈跟踪等
- 向用户显示友好的错误提示，避免暴露系统内部错误

#### 2.1.6 日志规范
- 使用统一的日志工具，如 `winston`（后端）或 `console`（前端）
- 日志级别：
  - `debug`：调试信息，仅在开发环境使用
  - `info`：一般信息，如用户登录、操作记录等
  - `warn`：警告信息，如 deprecated 功能使用、性能问题等
  - `error`：错误信息，如异常、API调用失败等
- 日志格式应包含：时间、级别、模块、消息、上下文信息等
- 避免在生产环境记录敏感信息，如密码、API密钥等

### 2.2 前端代码规范

#### 2.2.1 Vue组件规范
- 组件命名：使用大驼峰命名法，组件名应清晰表达其功能
  - 示例：`MeetingList`, `UserProfile`, `SpeechTranscript`
- 组件结构：
  ```vue
  <template>
    <!-- 模板内容 -->
  </template>

  <script setup lang="ts">
  // 组件逻辑
  </script>

  <style scoped>
  /* 组件样式 */
  </style>
  ```
- 使用 `<script setup lang="ts">` 语法编写组件逻辑
- 组件 props 应添加类型定义和默认值
- 组件事件应使用 `defineEmits` 定义
- 组件内部状态应使用 `ref` 或 `reactive` 定义
- 计算属性应使用 `computed` 定义
- 生命周期钩子应按照执行顺序使用
- 样式使用 `scoped` 修饰符，避免样式冲突
- 使用 CSS 预处理器（如 SCSS）时，遵循其最佳实践

#### 2.2.2 Vue Router规范
- 路由命名：使用小驼峰命名法，路由名应与组件名对应
  - 示例：`meetingList`, `userProfile`, `speechTranscript`
- 路由路径：使用短横线分隔的小写命名法
  - 示例：`/meeting-list`, `/user-profile`, `/speech-transcript`
- 路由配置应包含：path, name, component, meta等
- 权限控制应在路由 meta 中定义
- 路由懒加载：使用动态 import 加载组件，提高初始加载性能
  - 示例：`component: () => import('@/views/MeetingList.vue')`

#### 2.2.3 Pinia规范
- Store命名：使用小驼峰命名法，Store名应清晰表达其管理的状态
  - 示例：`userStore`, `meetingStore`, `speechStore`
- Store结构：
  ```typescript
  import { defineStore } from 'pinia'

  export const useUserStore = defineStore('user', {
    state: () => ({
      // 状态定义
    }),
    getters: {
      // 计算属性
    },
    actions: {
      // 方法
    }
  })
  ```
- 状态名应清晰表达其用途
- Getters应使用小驼峰命名法
- Actions应使用小驼峰命名法，动词开头
- 避免在Store中直接操作DOM

### 2.3 后端代码规范

#### 2.3.1 Express规范
- 路由命名：使用RESTful API设计原则，路径使用短横线分隔的小写命名法
  - 示例：`/api/v1/meetings`, `/api/v1/users/:id`, `/api/v1/speech/transcript`
- 路由配置：将路由按模块拆分，使用路由中间件
  - 示例：`app.use('/api/v1/meetings', meetingRoutes)`
- 中间件：按功能拆分，如认证中间件、日志中间件、错误处理中间件等
- 控制器：每个控制器负责处理一个模块的请求，方法名应与HTTP方法对应
  - 示例：`getMeetings()`, `createMeeting()`, `updateMeeting()`, `deleteMeeting()`
- 服务层：业务逻辑应封装在服务层，控制器仅负责请求处理和响应返回
- 错误处理：使用统一的错误处理中间件，返回标准化的错误响应

#### 2.3.2 数据库规范

##### 2.3.2.1 PostgreSQL规范
- 表命名：使用下划线分隔的小写命名法，表名应清晰表达其内容
  - 示例：`users`, `meetings`, `speakers`, `meeting_participants`
- 列命名：使用下划线分隔的小写命名法，列名应清晰表达其用途
  - 示例：`user_id`, `meeting_name`, `start_time`, `is_active`
- 主键命名：使用 `表名_id` 格式
  - 示例：`user_id`, `meeting_id`
- 外键命名：使用 `关联表名_id` 格式
  - 示例：`user_id`, `meeting_id`
- 索引命名：使用 `idx_表名_列名` 格式
  - 示例：`idx_users_email`, `idx_meetings_start_time`
- SQL语句：关键字使用大写，表名和列名使用小写
  - 示例：`SELECT * FROM users WHERE email = 'test@example.com'`

##### 2.3.2.2 MongoDB规范
- 集合命名：使用小驼峰命名法，集合名应清晰表达其内容
  - 示例：`speechRecords`, `aiAnalyses`, `meetingTranscripts`
- 字段命名：使用小驼峰命名法，字段名应清晰表达其用途
  - 示例：`userId`, `meetingId`, `speechText`, `timestamp`
- 索引命名：使用 `idx_字段名` 格式
  - 示例：`idx_userId`, `idx_meetingId`
- 文档结构：保持文档结构的一致性，避免频繁变更
- 嵌套层级：避免过深的嵌套，建议不超过3层

### 2.4 代码质量工具

#### 2.4.1 ESLint
- 前端使用 `@vue/eslint-config-typescript` 配置
- 后端使用 `@typescript-eslint/eslint-plugin` 配置

- 规则：遵循默认规则，根据项目需求调整
- 执行命令：`npm run lint`

#### 2.4.2 Prettier
- 配置文件：`.prettierrc.js`
- 规则：

  {

    "trailingComma": "es5",
    "tabWidth": 2,

    "printWidth": 120
  }

#### 2.4.3 TypeScript
- 配置文件：`tsconfig.json`
- 严格模式：开启严格模式，提高类型安全性
- 编译选项：根据项目需求配置，如 `target`, `module`, `outDir` 等

## 3. 开发流程

### 3.1 分支管理策略

#### 3.1.1 分支类型
- **main**：主分支，用于生产环境，仅接受来自 develop 分支的合并
- **develop**：开发分支，用于集成所有功能，接受来自 feature 分支的合并
- **feature/xxx**：功能分支，用于开发新功能，从 develop 分支创建，完成后合并回 develop 分支
- **bugfix/xxx**：bug修复分支，用于修复bug，从 develop 或 main 分支创建，完成后合并回原分支
- **hotfix/xxx**：紧急修复分支，用于修复生产环境的紧急bug，从 main 分支创建，完成后合并回 main 和 develop 分支
- **release/xxx**：发布分支，用于准备发布，从 develop 分支创建，完成后合并回 main 和 develop 分支

#### 3.1.2 分支命名规范
- **feature分支**：`feature/功能名称`
  - 示例：`feature/meeting-management`, `feature/speech-recognition`
- **bugfix分支**：`bugfix/问题描述`
  - 示例：`bugfix/fix-meeting-creation-error`, `bugfix/resolve-speech-transcript-delay`
- **hotfix分支**：`hotfix/问题描述`
  - 示例：`hotfix/fix-production-crash`, `hotfix/resolve-security-vulnerability`
- **release分支**：`release/版本号`
  - 示例：`release/1.0.0`, `release/1.1.0`

#### 3.1.3 分支操作流程
1. 从当前最新的 develop 分支创建 feature 分支
   ```bash
   git checkout develop
   git pull origin develop
   git checkout -b feature/xxx
   ```
2. 开发完成后，提交代码并推送到远程仓库
   ```bash
   git add .
   git commit -m "feat: 实现xxx功能"
   git push origin feature/xxx
   ```
3. 创建 PR，请求将 feature 分支合并到 develop 分支
4. 进行代码评审，通过后合并到 develop 分支
5. 删除本地和远程的 feature 分支
   ```bash
   git checkout develop
   git branch -d feature/xxx
   git push origin --delete feature/xxx
   ```

### 3.2 提交规范

#### 3.2.1 Conventional Commits 规范
使用 Conventional Commits 规范，格式如下：
```
<type>[optional scope]: <description>

[optional body]

[optional footer(s)]
```

#### 3.2.2 类型说明
- **feat**：新功能
- **fix**：修复bug
- **docs**：文档更新
- **style**：代码风格变更（不影响代码逻辑）
- **refactor**：代码重构（不新增功能，不修复bug）
- **test**：测试相关
- **chore**：构建过程或辅助工具的变动
- **perf**：性能优化
- **ci**：CI/CD配置变更
- **build**：构建配置变更
- **revert**：回滚到之前的提交

#### 3.2.3 示例
- 简单提交：
  ```
  feat: 实现会议创建功能
  ```
- 带作用域的提交：
  ```
  feat(meeting): 实现会议创建功能
  ```
- 带正文的提交：
  ```
  feat(meeting): 实现会议创建功能

  添加了会议创建表单，支持设置会议名称、时间、参与者等信息
  ```
- 带页脚的提交：
  ```
  fix(speech): 修复语音转录延迟问题

  优化了语音识别算法，减少了转录延迟

  closes #123
  ```

#### 3.2.4 提交工具
- 使用 `commitizen` 工具辅助创建符合规范的提交信息
- 配置 `husky` 和 `lint-staged` 进行提交前检查

### 3.3 代码评审流程

#### 3.3.1 评审准备
- 确保代码符合代码规范
- 确保所有测试通过
- 提供清晰的PR描述，包括功能说明、变更内容、测试情况等
- 关联相关的需求或issue

#### 3.3.2 评审标准
- 代码质量：是否符合代码规范，是否遵循最佳实践
- 功能完整性：是否实现了所有需求，是否存在遗漏
- 测试覆盖率：是否有足够的测试用例，测试覆盖率是否达标
- 性能：是否存在性能问题，是否有优化空间
- 安全性：是否存在安全漏洞，是否遵循安全最佳实践
- 可维护性：代码是否易于理解、维护和扩展

#### 3.3.3 评审流程
1. 创建PR，请求将分支合并到目标分支
2. 指派至少2名评审人员
3. 评审人员进行代码评审，提出意见和建议
4. 作者根据评审意见进行修改
5. 修改完成后，评审人员再次进行评审
6. 评审通过后，合并到目标分支
7. 删除临时分支

#### 3.3.4 评审反馈
- 反馈应具体、明确，避免模糊的描述
- 反馈应包含问题所在、原因和改进建议
- 反馈应尊重作者，使用建设性的语言
- 作者应积极响应评审反馈，及时进行修改

### 3.4 CI/CD流程

#### 3.4.1 持续集成
- **触发条件**：当代码推送到远程仓库或PR创建时触发
- **执行步骤**：
  1. 安装依赖
  2. 执行代码质量检查（ESLint, Prettier）
  3. 执行类型检查（TypeScript）
  4. 执行单元测试和集成测试
  5. 构建项目
- **成功条件**：所有检查和测试通过
- **通知方式**：通过邮件、Slack等方式通知结果

#### 3.4.2 持续部署
- **触发条件**：当代码合并到main分支时触发
- **执行步骤**：
  1. 安装依赖
  2. 执行代码质量检查
  3. 执行类型检查
  4. 执行所有测试
  5. 构建项目
  6. 部署到测试环境
  7. 执行E2E测试
  8. 部署到预发布环境
  9. 手动验证
  10. 部署到生产环境
- **成功条件**：所有检查、测试和部署步骤通过
- **回滚机制**：如果部署失败，自动回滚到上一个稳定版本

### 3.5 版本管理

#### 3.5.1 版本号格式
使用语义化版本号（Semantic Versioning），格式为：`MAJOR.MINOR.PATCH`
- **MAJOR**：不兼容的API变更
- **MINOR**：向下兼容的功能新增
- **PATCH**：向下兼容的bug修复

#### 3.5.2 版本发布流程
1. 从 develop 分支创建 release 分支
   ```bash
   git checkout develop
   git pull origin develop
   git checkout -b release/1.0.0
   ```
2. 更新版本号，修改相关配置文件
3. 提交代码并推送到远程仓库
   ```bash
   git add .
   git commit -m "chore: 更新版本号到1.0.0"
   git push origin release/1.0.0
   ```
4. 进行最终测试，确保所有功能正常
5. 合并 release 分支到 main 和 develop 分支
   ```bash
   git checkout main
   git merge --no-ff release/1.0.0
   git tag v1.0.0
   git push origin main --tags
   
   git checkout develop
   git merge --no-ff release/1.0.0
   git push origin develop
   ```
6. 删除 release 分支
   ```bash
   git branch -d release/1.0.0
   git push origin --delete release/1.0.0
   ```

### 3.6 发布流程

#### 3.6.1 发布准备
- 完成所有功能开发和bug修复
- 确保所有测试通过
- 更新文档，包括README、API文档、用户手册等
- 进行安全扫描，确保没有安全漏洞
- 进行性能测试，确保性能符合要求

#### 3.6.2 发布步骤
1. 创建 release 分支，更新版本号
2. 部署到测试环境，进行系统测试
3. 部署到预发布环境，进行用户验收测试
4. 收集反馈，修复发现的问题
5. 部署到生产环境
6. 发布版本公告，包括新功能、bug修复、变更说明等
7. 监控生产环境，确保系统稳定运行

#### 3.6.3 回滚流程
- 如果生产环境出现严重问题，立即执行回滚操作
- 回滚到上一个稳定版本
- 分析问题原因，修复后重新发布
- 通知相关人员，说明回滚原因和后续计划

## 4. 最佳实践

### 4.1 SOLID原则
- **单一职责原则**：每个类或函数只负责一个功能
- **开放封闭原则**：对扩展开放，对修改关闭
- **里氏替换原则**：子类可以替换父类，不影响系统功能
- **接口隔离原则**：使用多个小接口，避免使用大接口
- **依赖倒置原则**：依赖抽象，不依赖具体实现

### 4.2 设计模式
- **工厂模式**：用于创建对象，隐藏创建逻辑
- **单例模式**：确保一个类只有一个实例
- **观察者模式**：用于对象间的一对多依赖关系
- **策略模式**：用于定义算法家族，动态选择算法
- **装饰器模式**：用于动态扩展对象功能
- **适配器模式**：用于转换接口，使不兼容的类可以一起工作

### 4.3 性能优化
- **前端性能优化**：
  - 使用懒加载，减少初始加载时间
  - 优化图片资源，使用适当的格式和尺寸
  - 减少HTTP请求，使用HTTP/2和CDN
  - 优化JavaScript代码，避免不必要的计算
  - 使用缓存，减少重复请求
- **后端性能优化**：
  - 优化数据库查询，使用索引
  - 使用缓存，减少数据库访问
  - 优化API设计，减少数据传输
  - 使用异步编程，提高并发处理能力
  - 优化代码逻辑，减少不必要的计算

### 4.4 安全最佳实践
- **认证和授权**：
  - 使用JWT进行认证
  - 实现RBAC（基于角色的访问控制）
  - 定期更新密码和密钥
  - 实现密码重置功能
- **数据保护**：
  - 加密敏感数据，如密码、API密钥等
  - 使用HTTPS传输数据
  - 实现数据备份和恢复机制
  - 限制数据访问权限，遵循最小权限原则
- **输入验证**：
  - 对所有用户输入进行验证
  - 防止SQL注入、XSS攻击等
  - 使用参数化查询，避免直接拼接SQL语句
- **错误处理**：
  - 不要向用户暴露系统内部错误
  - 记录详细的错误日志，便于调试
  - 实现错误监控和告警机制

### 4.5 测试最佳实践
- **测试驱动开发（TDD）**：先编写测试用例，再实现功能
- **单元测试**：测试最小的可测试单元，如函数或方法
- **集成测试**：测试不同模块之间的交互
- **E2E测试**：测试整个系统的端到端功能
- **性能测试**：测试系统在不同负载下的性能表现
- **安全测试**：测试系统的安全性，包括认证、授权、数据保护等
- **测试覆盖率**：确保测试覆盖足够的代码和功能
- **自动化测试**：尽可能自动化测试，提高测试效率和一致性
